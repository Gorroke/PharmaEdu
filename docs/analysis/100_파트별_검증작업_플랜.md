## 작업 단위 (1 파트당 1주기)

```
1) 분석 (1~2명 병렬, read-only)
   ├─ 우리 코드 vs 윈팜 차이 case-by-case 비교
   ├─ "우리 버그" / "윈팜이 다름" / "표준 모름" 분류
   └─ HIRA 작성요령 PDF 인용 (필요 시)
        ↓
2) 사용자 보고 + 결정 (분석가 → 사용자)
        ↓
3) 수정 (사용자 승인 후, 1명만)
   ├─ 코드 수정
   ├─ 회귀 19/19 확인
   └─ TS 컴파일 0 확인
        ↓
4) 윈팜 재검증 (자동)
   └─ verify.ts 재실행 → 해당 카테고리 일치율 변화 보고
        ↓
5) 다음 파트로 진입
```

각 파트마다 **회귀 19/19 + 윈팜 일치율 상승** 둘 다 통과해야 다음 파트.

## 파트 정의 (9개, calc-engine 데이터 흐름 순서)

### P1. 약품금액 (drug-amount.ts)
- **범위**: 단가 × 1회량 × 일수 × 횟수, Del_Yn, EXTYPE 필터, 648 ForceSetSum, sectionTotals 분류
- **파일**: `src/lib/calc-engine/drug-amount.ts`, `src/lib/calc-engine/types.ts`(DrugItem)
- **윈팜 패턴 연결**: (직접 발견 없음)
- **검증 카테고리**: CAT01/02/03, CAT45 (단순 계산)
- **현재 일치 상태**: ✅ 100% (CAT01/02/03/45 등)
- **상태**: 확인만 필요 (코드 변경 없을 가능성)

### P2. 조제료 기본 (dispensing-fee.ts)
- **범위**: Z1000~Z5000 기본 코드, 25구간 의약품관리료(Z5101~Z5391), 처방조제 분기, dosDate 8자리
- **파일**: `src/lib/calc-engine/dispensing-fee.ts`, `supabase/seed.sql`
- **윈팜 패턴 연결**: 일부 (CAT47/48/49/50 투약일수 차이)
- **검증 카테고리**: CAT46~CAT50 (투약일수)
- **현재 일치 상태**: 1일 100%, 7일/15일/30일/60일 부분 FAIL
- **확인할 것**: 25구간 가산이 윈팜 단가와 일치하는가, dosDate 분기

### P3. 가산 (surcharge.ts + dispensing-fee.ts) ⭐ **여기서 시작 권장**
- **범위**: 야간/토요/공휴일/명절/심야/소아 가산, text3 차등수가, Z4121 외용 병용, 자가투여 Z4130
- **파일**: `src/lib/calc-engine/surcharge.ts`, `dispensing-fee.ts` (가산 부분)
- **윈팜 패턴 연결**:
  - **Pattern A 토요/공휴일 가산** ⭐ **확실한 우리 버그**
  - **Pattern C Z4121 외용** ⭐ **확실한 우리 버그/데이터 부족**
  - **Pattern F 차등수가** (191,877원 차이, 양쪽 의심)
- **검증 카테고리**: CAT14/15/19/44 토요공휴 (12 FAIL), CAT08/10/51/53 보훈+Z코드 (6 FAIL), CAT38/40 차등수가 (5 FAIL)
- **현재 일치 상태**: 0~33%
- **예상 효과**: P3 끝나면 **18~23건 PASS 추가** (전체 일치율 49% → 64%+)

### P4. 보험 유형별 분기 (copayment.ts 입구)
- **범위**: C/D/G/F/E/B 분기, 처방조제/직접조제 분기, 서식번호 결정 (H024/H124)
- **파일**: `src/lib/calc-engine/copayment.ts` 진입부, `index.ts`
- **윈팜 패턴 연결**: (간접) — 분기 자체는 정확하나 하위 로직 영향
- **검증 카테고리**: 모든 보험구분 카테고리 진입점

### P5. 본인부담률 결정 (copayment.ts + medical-aid.ts)
- **범위**: 65세/6세/75세 연령 분기, 산정특례 V103/V206/V246/V252, 의료급여 sbrdnType (M코드/B코드), B014 정률
- **파일**: `src/lib/calc-engine/copayment.ts`, `modules/insurance/medical-aid.ts`
- **윈팜 패턴 연결**:
  - **Pattern E 산정특례 요율** (V103/V252/V206/V246, 양쪽 의심)
  - **CAT16 6세 미만**, **CAT07 D80**
- **검증 카테고리**: CAT16/30/31/33, CAT05/06/07
- **확인할 것**: V코드 요율이 HIRA 고시와 일치하는가

### P6. 본인부담금 계산 + 선별급여 (copayment.ts)
- **범위**: 정률/정액 적용, 선별급여 A/B/D/E항 독립 계산, U항 + totalPrice2
- **파일**: `src/lib/calc-engine/copayment.ts` (B-2~B-3 구현 부분)
- **윈팜 패턴 연결**:
  - **Pattern B U항 totalPrice 정의** ⭐ **표준 재검토 필요**
  - **CAT26 100%자부담** (3 FAIL, +17,147원 평균)
- **검증 카테고리**: CAT26/05 일부
- **확인할 것**: HIRA 작성요령 PDF에서 totalPrice/totalPrice2 정확한 정의 추출 → 우리 B-3 재정의

### P7. 보훈 (veteran.ts)
- **범위**: bohunCode M10~M90 분기, G10/G20 두 청구체계, GsCode/MT038, calcMpvaPrice 필터, MpvaComm
- **파일**: `src/lib/calc-engine/modules/insurance/veteran.ts`, `gs-code.ts`
- **윈팜 패턴 연결**:
  - **CAT13 B024** (보훈 G10 + B024, 우리 0원 vs 윈팜 28,500원)
  - 보훈 세부 M코드는 윈팜에 컬럼 없어 직접 검증 불가
- **검증 카테고리**: CAT13, CAT08/10/51/53 일부
- **확인할 것**: B024 코드가 무엇인지, 우리 보훈 진입 조건이 정확한가

### P8. 3자배분 + 공비 (copayment.ts 후반)
- **범위**: pubPrice/insuPrice/realPrice, 특수공비 302/101/102 (B-4/B-5)
- **파일**: `src/lib/calc-engine/copayment.ts` (B-4/B-5 부분)
- **윈팜 패턴 연결**: 직접 패턴은 없음 (Phase B에서 이미 항등식 패치 완료)
- **확인할 것**: 보훈/F10 등에서 3자배분 정확성 재확인

### P9. 반올림 + 결과 조립 (rounding.ts + index.ts)
- **범위**: trunc10/trunc100/round1/ceil10, NPayRoundType 6종, 서식번호 / formNumber
- **파일**: `src/lib/calc-engine/rounding.ts`, `index.ts`
- **윈팜 패턴 연결**: (간접) — 1원 단위 차이 흔적
- **확인할 것**: 우리 반올림 함수가 윈팜과 동일한 시점에 적용되는가

## 시작점

**P3 가산부터 시작**합니다. 이유:

1. **윈팜 검증으로 명확한 우리 버그가 식별됨** (Pattern A 토요030 단가, Pattern C Z4121 외용)
2. **즉시 효과 큼** — P3 끝나면 18~23건 FAIL → PASS 예상 (일치율 49% → 64%+)
3. **위험 낮음** — Pattern A는 한 함수 한 줄 추가 수준
4. **분석 시간 절약** — 이미 검증 데이터로 어디가 문제인지 보임

P1/P2는 100% 일치 상태이므로 **검증만** (분석 단계만 수행, 코드 변경 없을 가능성). P3에서 효과 확인 후 P4~P9 순차 진행.

## 작업 단계 (P3 시작 시)

1. **P3 분석가 1명 투입** — read-only로 다음 분석:
   - `surcharge.ts`, `dispensing-fee.ts` 가산 부분 정독
   - Pattern A: `applySaturdaySurchargeRows` 함수 정확한 라인 확인
   - Pattern C: Z4121 외용 분류가 어디서 어떻게 깨지는지 확인
   - Pattern F: 차등수가 191K원 차이의 정확한 원인 (우리 vs 윈팜)
   - 사용자 보고용 분석 보고서 작성 (`docs/analysis/P3_분석.md`)

2. **사용자 보고 + 결정**
   - 어떤 게 우리 버그, 어떤 게 윈팜 차이, 어떤 게 추가 검증 필요
   - 수정 범위 확정

3. **수정 1명 투입** (사용자 승인 후)
   - Pattern A fix → 회귀 → 윈팜 재검증
   - Pattern C fix → 회귀 → 윈팜 재검증
   - Pattern F는 분석 결과에 따라

4. **상태 업데이트**
   - 본 문서의 진행 상태 표 갱신
   - 일치율 변화 기록
