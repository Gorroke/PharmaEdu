## 1. 소스 프로젝트 개요

CH00~CH12 분석 문서는 다음 4개 소스를 교차 참조한다. 각 소스의 성격·기술스택·신뢰도를 이해해야 소스 간 차이를 올바르게 해석할 수 있다.

### 1.1 비즈팜 (BizPharm) — 약칭 `[비즈팜]` / `BIZ`

| 항목 | 내용 |
|------|------|
| **공식 명칭** | 비즈팜 (BizPharm) |
| **기술 스택** | VB6 (Visual Basic 6) |
| **핵심 파일** | `Ppre1000__.frm` |
| **언어/타입 시스템** | VB6 `Currency` / `Double` — IEEE 754 부동소수점 사용 |
| **아키텍처 특징** | UI 중심(Form). 계산 로직과 UI가 분리되지 않음. 스프레드시트(sprDrug) 행 순회 방식으로 약품 처리 |
| **DB 방식** | 직접 테이블 조회 (`PPREVALU`, `PDRUDRUG`, `PDRUVALU`, `PZCVALUE`, `PCOMDAY1` 등) |
| **참조 위치** | CH01, CH03, CH04, CH05, CH06, CH07, CH08, CH09, CH10, CH12 |
| **신뢰도 / 알려진 문제** | - VB6 `Double`로 인한 부동소수점 오차 위험<br>- UI와 계산 로직이 혼재 → 순수 계산 규칙 추출이 어려움<br>- `Del_Yn` 분기(A/B/C/F/G/M/P)가 타 소스에 없는 7가지 복잡한 분기<br>- 선별급여(A/B/D/E) 미지원: "부분" 1개 범주로만 처리<br>- 2016.09.29 절사 변경 분기 없음 (C31/C32 날짜 분기 누락)<br>- CH08에서 버그 사례 다수 확인 — 새 시스템에서 반복 금지 |

| 항목 | 내용 |

| **신뢰도** | **최고 권위** — 법적 청구 기준이므로 다른 소스와 충돌 시 우선 적용 원칙. 단, PDF에 명시되지 않은 세부 분기는 코드 소스 3종에서 보완 |

## 2. 소스 차이점 카탈로그

CH01과 CH07에서 식별된 모순/충돌을 항목별로 정리한다. 각 항목에는 **4소스가 어떻게 다른지**만 서술하며, 구현 선택은 §4(구현 입장 프레임워크)에서 다룬다.

---

### M01. 단가 결정 로직 (CH01-M01)

**무엇이 다른가**: 단가를 어디서 가져오는가, 언제 결정하는가가 소스별로 근본적으로 다르다.

| 소스 | 단가 소스 | 결정 시점 | 특이사항 |
|------|----------|-----------|---------|
| **비즈팜** | `PPREVALU.insurance_amt` → `PDRUDRUG.ilban_amt` → `PDRUVALU.Amt4~1` (3단계 폴백 체인) | DB 조회 시 유효기간 범위 내 결정 | `Del_Yn` 7가지 분기(A/B/C/F/G/M/P) 포함 |
| **공단** | "약제 및 치료재료의 비용에 대한 결정기준에 따른 단가" | 명시 없음 (외부 마스터 전제) | 산식 정의만 있고 단가 결정 절차는 없음 |
| **유팜** | 보험단가 / 실거래가 / 판매단가 (3분기) | `GetDrugPrice()` 호출 시 동적 결정 | `WAPPid>0`이면 실거래가 비교 후 저가 선택 |
| **EDB** | `PD_PRICE` 단일 필드 | `DrugItem` 생성 시 이미 확정 (외부에서 주입) | 엔진 내부에 단가 결정 로직 없음 |

---

### M02. 비보험 분류 — 전액본인부담 귀속 (CH01-M02)

**무엇이 다른가**: `전액본인부담` 약품을 비급여 합계에 포함할지 별도 항목으로 분리할지가 소스별로 다르다.

| 소스 | 비급여 필터 조건 | 전액본인부담 귀속 |
|------|----------------|----------------|
| **유팜 직접조제** | `!보험 \|\| 전액본인부담` | 비급여에 **포함** |
| **유팜 처방조제** | `!보험 && !전액본인부담` | 비급여에서 **제외** (별도 `Calculate전액본인부담약가`) |
| **비즈팜** | `TypeButtonText = "비"/"비급여"/"부분"` | "부분"으로 분류 시 감면율 적용 |
| **EDB** | `PD_INSUPAY = "0"` (비급여), `"9"` (100%본인) | `PD_INSUPAY="9"` 별도 집계 |
| **공단** | U항 = 100/100 본인부담 (p.648~655) | 항상 별도 U항으로 분리 |

---

### M03. 수량(소모량) 계산 순서 — 팩 처리 및 부동소수점 (CH01-M03)

**무엇이 다른가**: 팩 나누기를 언제 하는가, 어떤 타입으로 계산하는가가 다르다.

| 소스 | 기본 공식 | 팩 처리 방식 | 타입 |
|------|----------|------------|------|
| **비즈팜** | `Int(One_Qty * IN_Qty * Tot_Cnt * Danga + 0.5)` — 한 번에 계산 | 팩 개념 미확인 | VB6 Double (오차 위험) |
| **공단** | `단가 × 1회투약량 × 1일투여횟수 × 총투여일수`, 원미만 4사5입 | 팩단위: 1회투약량을 팩 분모로 환산 (p.569) | 명시 없음 |
| **유팜** | `(단가+장려금) × (1+할증/100) × Get소모량()` → `To원미만사사오입()` | 팩>1이면 사전 계산된 소모량 사용 | C# decimal |
| **EDB** | `(int)(DOSE * DNUM * DDAY / PACK * PRICE + 0.5m)` | 소모량 계산 중간에 `/PACK` 수행 후 `*PRICE` | C# decimal |

추가 차이: 유팜만 중간 반올림(`To원미만사사오입`)을 적용한다. 비즈팜은 `Int(x+0.5)`, EDB는 `(int)(x+0.5m)` — 음수 영역에서 다를 수 있으나 금액은 항상 양수이므로 실질 차이 없음.

---

### M04. 선별급여 (A/B/D/E항) 처리 (CH01-M04)

**무엇이 다른가**: 비즈팜은 선별급여 A/B/D/E 구분 자체가 없다.

| 소스 | A/B/D/E 개념 | 합산 방식 |
|------|-------------|---------|
| **비즈팜** | **없음** — "부분" 1개 범주로 통합. 감면율은 `cboBoHum5` 드롭다운(60%/50%/30%)으로 결정 (보훈 감면율, 선별급여 요율 아님) | `Insur_Non_Amt`에 비율 곱한 값 누적 |
| **공단** | A항 50% / B항 80% / D항 30% / E항 90% (p.152~153, p.486~487) | 각 항별 금액 합산 후 개별 요율 적용, 합산 후 10원 절사 |
| **유팜** | `Calculate백대백미만약가A/B/D/E` 4개 별도 메서드 | 각 `백대백미만Types`별 별도 `CalcResult` |
| **EDB** | `PD_INSUPAY="2"(50%)`, `"3"(80%)`, `"4"(30%)`, `"5"(90%)` 별도 집계 | `m_pb_SumInsuDrug_50/80/30/90` 개별 변수 |

---

### M05. 사용장려금 (퇴장방지의약품) 적용 범위 (CH01-M05)

**무엇이 다른가**: 공단은 직접조제만 가능하다고 명시하지만, 유팜은 처방조제에도 관련 계산이 존재한다. EDB·비즈팜은 코드에서 미확인.

| 소스 | 대상 범위 | 처방조제 처리 | 단가 산식 |
|------|----------|------------|---------|
| **공단** | 의약분업 예외지역 **직접조제만** (p.569~570) | **산정 불가** (명시) | `round(상한금액 × 10%)`, 최소 1원 |
| **유팜** | 직접조제: `저가가산금 × 소모량`, 처방조제: `저가대체조제장려금` 별도 처리 | 처방조제에서도 `저가대체조제장려금` 계산 존재 | 외부 마스터에서 `저가가산금` 주입 |
| **EDB** | 코드에서 전용 로직 미확인 | - | - |
| **비즈팜** | 코드에서 전용 로직 미확인 | - | - |

주의: "퇴장방지의약품 사용장려금"과 "저가대체조제 장려금"은 별개 제도다. 유팜의 처방조제 계산이 전자인지 후자인지 구분이 필요하다.

### M06. 코로나19 치료제 처리 (CH01-M06)

**무엇이 다른가**: 유팜은 금액을 50,000원으로 고정하고, EDB는 정상 약가에 5% 가산한다. 두 접근은 서로 다른 정책 측면을 구현한 것으로 추정된다.

| 소스 | 적용 기준일 | 처리 방식 | 금액 산식 |
|------|-----------|---------|---------|
| **유팜** | `코로나19_본인부담_2024적용일` 이후 | `drug.금액 = 50,000원` 고정 (`!의료급여 && !차상위` 조건) | 무조건 50,000원 고정 (본인부담 총액 관점 추정) |
| **EDB** | `20241025` 이후 | `648903860` 코드 약품에 `PD_DDAY` max 5일 제한 + 약품합계의 5% 가산 | 정상 약가 계산 후 `num10 = num9 * 0.05` (보험 약가 가산 관점 추정) |
| **비즈팜** | 미확인 | 미확인 | - |
| **공단** | 미확인 | 미확인 | - |

### M07. 수가 단가 반올림 방식 (CH07-M01)

### M10. 유팜 MathHelper 명칭 버그 (CH07-M04)

| ID | 항목 | 심각도 | 4소스 차이 요약 |

## 부록 B. 충돌 우선순위 요약 (빠른 참조)

챕터 분석자가 구현 입장을 결정할 때 참고할 우선순위 원칙:

1. **공단(HIRA) 문서에 명시된 규칙** → 최우선 적용
2. **유팜+EDB 일치** (비즈팜 불일치) → 비즈팜은 오래된 VB6 구현이므로 유팜/EDB 기준 우선
3. **유팜 vs EDB 불일치** → 별도 법령/고시 원문 확인 필요 (M06 코로나19, M09 보훈 반올림 등)
4. **비즈팜만 다름** → 신규 구현에서 비즈팜 방식 지양 (M04 선별급여, M10 명칭 등)
5. **날짜 분기 (EDB만 존재)** → 고시 원문 확인 후 필요 여부 판단 (M08 의료급여, M11 C31/C32)
