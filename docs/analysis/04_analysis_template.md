## 1. 분석가용 템플릿

## 3. 챕터 요구사항 vs 우리 구현

> 챕터의 각 요구사항(§ 단위)을 행으로 나열하고, 우리 구현과의 매핑 상태를 표시한다.  
> **상태 기호**: ✓ 완전 구현 | ⚠ 부분 구현 또는 의심 | ✗ 미구현

| 챕터 §      | 요구사항 요약                     | 우리 구현 위치                              | 상태    | 비고 |
|------------|----------------------------------|------------------------------------------|---------|------|
| CH{N} §1.1 | {요구사항 한 줄 요약}             | `src/lib/xxx.ts:funcName():L100`         | ✓       |      |
| CH{N} §1.2 | {요구사항 한 줄 요약}             | `src/lib/xxx.ts:funcName():L200`         | ⚠       | 엣지케이스 미커버 |

## 3. 심각도 등급 정의

Phase 8 전체에서 통일하여 사용하는 4단계 심각도 체계이다.

| 심각도 | 기호 | 정의 | 예시 |
|-------|------|------|------|
| **Critical** | 🔴 | 잘못 계산되어 청구 반송 또는 법적 문제가 발생할 수 있음. 즉각적인 수정이 필요하며 릴리스 전에 반드시 해결해야 한다. | 본인부담금 계산 공식 자체가 틀림, 날짜 기준 분기 완전 누락 |
| **High** | 🟠 | 특정 조건에서 1원 이상의 금액 차이가 발생할 수 있음. 실 사용자에게 영향을 미치나 모든 케이스에서 발생하지는 않는다. | 특정 보험 코드 처리 누락, decimal 정밀도 오차 |
| **Medium** | 🟡 | 일반적인 케이스에서는 올바르게 동작하나 엣지케이스를 커버하지 못함. 단기적 영향은 낮으나 중장기적으로 수정이 필요하다. | 극소수 케이스 분기 미처리, null 체크 누락 |
| **Low** | 🟢 | 계산 결과에는 영향이 없으나 코드 품질, 가독성, 미래 확장성에 영향을 미침. 여유가 있을 때 개선한다. | 변수명 불명확, 중복 코드, 미사용 상수 |

> **집계 우선순위**: 🔴 → 🟠 → 🟡 → 🟢 순서로 처리한다.  
> **집계자 참고**: Critical·High 항목만 Phase 8 최종 보고서의 "즉시 수정 목록"에 포함한다.

## 4. 이슈 분류 체계

Phase 8 집계자가 최종 보고서를 작성할 때 사용하는 3개 카테고리이다. 분석가와 검증자는 각 이슈를 반드시 아래 세 카테고리 중 하나로 분류해야 한다.

### 4-1. 누락 (Missing)

**정의**: 챕터 규격에서 요구하는 계산 로직, 분기, 처리 흐름이 우리 코드베이스에 **전혀 구현되어 있지 않은** 경우.

**라벨 표기 방법**:

```
[🔴 Missing / Critical] 항목명: 설명 (CH{N} §{섹션})
[🟠 Missing / High] 항목명: 설명 (CH{N} §{섹션})
```

**판단 기준**:
- 해당 기능의 함수·클래스·모듈이 코드베이스에 존재하지 않음
- 관련 코드가 주석 처리되거나 TODO로만 남아 있음

---

### 4-2. 의심 (Suspicious)

**정의**: 구현은 존재하나 계산 로직의 **정확도가 의심**되는 경우. 원본과 다른 수식 사용, 잘못된 상수, 분기 순서 오류 등이 해당한다.

**라벨 표기 방법**:

```
[🔴 Suspicious / Critical] 항목명: 의심 이유 (우리 구현 위치 vs 원본 참조)
[🟠 Suspicious / High] 항목명: 의심 이유 (우리 구현 위치 vs 원본 참조)
```

**판단 기준**:
- 원본 C# 수식과 TypeScript 수식이 다름
- 4소스 간 불일치 상황에서 근거 없이 특정 소스를 따름
- 상수값·계수·반올림 방식이 원본과 상이함

---

### 4-3. 부족 (Insufficient)

**정의**: 구현은 존재하고 핵심 로직은 맞으나 **케이스 커버리지가 부족**한 경우. 특정 조건, 엣지케이스, 날짜 분기 등이 빠진 경우이다.

**라벨 표기 방법**:

```
[🟠 Insufficient / High] 항목명: 어떤 케이스가 빠져 있는지 (우리 구현 위치)
[🟡 Insufficient / Medium] 항목명: 어떤 케이스가 빠져 있는지 (우리 구현 위치)
```

**판단 기준**:
- 주된 흐름은 구현되었으나 조건 분기 일부가 없음
- 특수 보험 코드, 특정 날짜 이전/이후 처리 등이 일반 케이스에만 적용됨
- null·0·음수 등 경계값 처리가 정의되지 않음

## 5. 교차 참조 표기 규칙

보고서 내 모든 참조는 아래 형식을 **반드시** 따른다. 일관된 형식이어야 집계 스크립트가 자동으로 파싱할 수 있다.

### 5-1. 챕터 섹션 참조

```
CH{챕터번호} §{섹션번호}
```

**예시**:

```
CH05 §3.2
CH12 §1
CH09 §4.1.2
```

- 챕터 번호는 2자리 숫자로 표기한다 (예: `CH05`, `CH12`).
- 섹션 번호는 원문 그대로 표기하되 숫자·점 이외의 문자는 생략한다.

---

### 5-2. 우리 코드 참조

```
src/{하위경로}/{파일명}.ts:{함수명}():L{줄번호}
```

**예시**:

```
src/lib/calc-engine/copayment.ts:calcCopay():L120
src/lib/calc-engine/copayment.ts:L120
src/components/PharmacyCalc.tsx:handleSubmit():L45
```

- 줄 번호는 분석 시점의 번호를 기재하며, 범위가 있을 때는 `L120-L145` 형식을 사용한다.
- 함수 컨텍스트 없이 줄 번호만 참조할 때는 `파일명.ts:L{줄번호}` 형식을 사용한다.

---

### 5-3. 원본 C# 참조

```
{파일명}.cs:{클래스명}.{함수명}():L{줄번호}
```

**예시**:

```
CopaymentCalculator.cs:CopaymentCalculator.CalcCopay_C():L450
PharmacyFeeCalc.cs:PharmacyFeeCalc.CalcDispFee():L200
CopaymentCalculator.cs:L450
```

- 함수명이 명확하지 않을 때는 클래스명과 줄 번호만으로도 허용한다.
- 클래스명이 파일명과 동일하면 클래스명은 생략 가능하다.

---

## 6. 작성 지침 및 주의사항

### 분석가(Analyst) 지침

1. **템플릿 그대로 사용**: 섹션 번호와 제목을 임의로 추가·삭제·변경하지 않는다.
2. **항목이 없을 때**: "없음"이라고 명시한다. 섹션 자체를 삭제하지 않는다.
3. **§ 번호 정확성**: 챕터 섹션 번호는 원문과 정확히 일치해야 한다.
4. **상태 기호 통일**: 매핑 테이블의 상태는 ✓ / ⚠ / ✗ 세 가지만 사용한다.
5. **미구현 영역 명시**: 섹션 2의 "미구현 영역"은 반드시 기재한다. 없으면 "없음"으로 표시.

### 검증자(Verifier) 지침

1. **독립성 유지**: 분석가 보고서를 검토하되, 분석가의 판단을 그대로 따르지 않는다. 독립적으로 코드와 원문을 검토한다.
2. **4소스 모두 확인**: 비즈팜·공단·유팜·EDB 중 하나라도 방식이 다르면 반드시 테이블에 기재한다.
3. **C# 원본 필수 확인**: TypeScript 코드만 보지 말고, 반드시 대응하는 C# 원본과 비교한다.
4. **타입 안전성**: TypeScript의 `number` 타입을 사용하는 금액 계산이 있으면 모두 5-1절에 기재한다.
5. **분기 체크리스트**: 날짜 분기(특히 2018-01-01, 2020-01-01 등 주요 법령 시행일), 보험 코드 분기, 특수 케이스 분기를 빠짐없이 확인한다.

### 공통 지침

- **실제 분석만 기재**: 추측이나 가정은 "추정:" 접두어를 붙여 구분한다.
- **라인 번호 검증**: 참조 줄 번호는 실제 코드를 확인하여 기재한다. 추정 번호는 `L~{줄번호}` 형식으로 표기한다.
- **완료 처리**: 보고서 상단의 `상태` 체크박스를 완료 시 `[x] 완료`로 변경한다.
- **집계자 연락**: 🔴 Critical 항목 발견 시 집계자에게 즉시 별도 통보한다.
